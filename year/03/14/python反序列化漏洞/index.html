python反序列化漏洞

python序列化的方式

python序列化用的是pickle库，（也有用到cpickle库的）。

import pickle
import pickletools

class student():
    def __init__(self):
        self.name = 'test'
        self.grade = 666

a = student()
s = pickle.dumps(a)
print(s)

b'\x80\x04\x959\x00\x00\x00\x00\x00\x00\x00\x8c\x08__main__\x94\x8c\x07student\x94\x93\x94)\x81\x94}\x94(\x8c\x04name\x94\x8c\x04test\x94\x8c\x05grade\x94M\x9a\x02ub.'

​ 如上就是python序列化数据的基本方式，输出的一大堆看不懂的字符就是序列化后的数据。python反序列化用到的是pickle.loads()方法，调用 Unpickler类来实现反序列化。

​ 而unpickle所有的数据操作都在栈上，上面的一大串序列化的数据通过unpickle读入后，会被翻译成一堆机器可以看懂的指令操作，通过这些操作来告诉机器应该怎样处理这些数据。利用pickletools可以反汇编这个序列化后的数据，查看具体进行了什么操作。

可以看到下面输出的很像汇编指令了，可以清晰的看到每条指令具体做了什么，然后我们还可以用

pickletools.optimize()来优化这些指令，去除掉一些不必要的指令。

然后来分析这些指令具体做了什么。（参考知乎 阮行止师傅的）

​ 首先字符串的第一个字节是\x80，机器看到这个操作符，就会再读一个字符串，读到了\x03，它告诉机器这是一个依据3号协议序列化的字符串（字符串序列化的协议有0、2、3、4这四个版本，默认是3号协议），然后这部分操作结束。

​ 然后读入c操作符，它连续读入两个字符串 module和name，规定以\n为分割，然后把module.name压进栈中，在这里也就是把 _ _ main _ _.student压进栈中。

​ 然后读到了）这个操作符，它的作用是”把一个空的tuple压入当前栈“。处理完这个操作符之后读到了\x81这个操作符，它的作用是：从栈中弹出一个元素记为 args；再弹出一个元素记为cls。接下来，执行cls._ _ new _ _(cls,*args)。意思就是从栈中弹出一个参数和一个class，然后利用这个参数实例化class，把得到的实例压入栈中。

​ 然后读入一个}，它的意思是”把一个空的dict压入栈中“，然后是mark操作符，它的作用是”把当前栈这个整体作为一个list，压进前序栈，把当前栈清空。“

​ 然后是MARK之后的内容，下一个操作符是X，它的意义是：读入一个字符串，以\n为结尾，然后把这个字符串压入栈中，这里的M操作符也是一样的意思，在这四个元素都压入栈之后，当前栈里面的元素是name,test,grade,666，前序栈里的内容是一个空的student实例以及一个空的dict。

​ 然后遇到了u操作符，它将当前栈的所有内容放入一个数组arr，然后把当前栈恢复到MARK时的状态，执行完成后，arr=[‘name’,‘test’,‘grade’,666]；当前栈里是一个空的student实例和一个空的dict。然后拿出dict，两个一组的读入arr里的元素，一个作为key，一个作为value。当前栈里的元素就变成了一个空的student实例，和一个dict[‘name’:‘test’,‘grade’:666]

​ 下一个指令码是b，就是BUILD指令，它把当前栈栈顶存入state，然后弹掉，把当前栈栈顶记为inst，然后弹掉。利用state这一系列的值来更新实例inst。把得到的对象放入当前栈。

​ 虽然我也不知道上面到底说了些啥，但是最后当前栈里面只剩下了一个实例student，并且拥有我们传入的属性和值。

python反序列化漏洞的出现 reduce

网上最多的介绍python反序列化漏洞的成因都是来自 _ _ reduce _ _方法，它的指令码是R。利用的方式很简单，只要在正常的类里面加上 _ _ reduce _ _ 方法，并且方法内构造一个恶意的指令。

这样序列化出来的数据，我们可以根据这个指令来自己构造

为什么在序列化的字符串中会出现q，其实是因为optimize把q指令码（也就是put给省略了）

这样就可以看见指令中有q了，但是这个q可有可无，无论加不加都可以执行命令。

如图直接去掉q指令码，也是可以执行命令的。

全局变量包含 c指令码

这题是出现在攻防世界战疫赛上的，一开始我一直想着rce，没想到可以使用变量覆盖。

原题大概是这样的，这里少了一个判断module是否为main的条件，不过无关紧要，这里直接看解法。传入payload

b'\x80\x03c__main__\nsecret\n}(Vname\nVrua\nVgrade\nVwww\nub0c__main__\nStudent\n)\x81}(X\x04\x00\x00\x00nameX\x03\x00\x00\x00ruaX\x05\x00\x00\x00gradeX\x03\x00\x00\x00wwwub.'

根据前面的指令内容可以知道，先将secret引入，然后压入一个dict内容为我们自定义的，然后执行BUILD指令，改写了原本题目中引入的secret为我们传入的值，弹掉栈顶，再传入一个正常的Student序列化之后的字符串，内容也是我们自定义。

所以name和grade值就相等了。

在我的实验过程中发现，如果是自己序列化构造的数据，构造出的指令码并不是V，而是X，且会有读入tuple和创建对象实例的操作，在修改题目中的secret值的时候我们是不需要这两个操作的，所以可以手动去掉，也就是说，下一次题目得自己手搓序列化后的数据。然后用pickletools构造器查看是否合法就行了。

原理就是阮行止师傅说的这样

这里的实现方式也就是上文的注所提到的：如果inst拥有__setstate__方法，则把state交给__setstate__方法来处理；否则的话，直接把state这个dist的内容，合并到inst.__dict__ 里面。

　　它有什么安全隐患呢？我们来想想看：Student原先是没有__setstate__这个方法的。那么我们利用{'__setstate__': os.system}来BUILE这个对象，那么现在对象的__setstate__就变成了os.system；接下来利用"ls /"来再次BUILD这个对象，则会执行setstate("ls /") ，而此时__setstate__已经被我们设置为os.system，因此实现了RCE.

上面的代码照抄一下就是

b'\x80\x03c__main__\nStudent\n)\x81}(V__setstate__\ncos\nsystem\nubVcalc.exe\nb.'

成功弹出计算器，但是后面会报错，因为没有传入Student属性，所以可以直接在后面加上。

payload

payload = b'\x80\x03c__main__\nStudent\n)\x81}(V__setstate__\ncos\nsystem\nubVls /\nb0c__main__\nStudent\n)\x81}(X\x04\x00\x00\x00nameX\x03\x00\x00\x00ruaX\x05\x00\x00\x00gradeX\x03\x00\x00\x00wwwub.'
