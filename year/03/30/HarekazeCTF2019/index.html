今天做了HarekazeCTF2019的两道Avatar Uploader，感觉非常有意思，于是在此记录一下。题目都是给了源码的，所以考验代码审计能力。

0x01 Avatar Uploader1

第一题比较简单，主要考的是两个函数对文件判断的区别。

<?php
error_reporting(0);

require_once('config.php');
require_once('lib/util.php');
require_once('lib/session.php');

$session = new SecureClientSession(CLIENT_SESSION_ID, SECRET_KEY);

// check whether file is uploaded
if (!file_exists($_FILES['file']['tmp_name']) || !is_uploaded_file($_FILES['file']['tmp_name'])) {
  error('No file was uploaded.');
}

// check file size
if ($_FILES['file']['size'] > 256000) {
  error('Uploaded file is too large.');
}

// check file type
$finfo = finfo_open(FILEINFO_MIME_TYPE);
$type = finfo_file($finfo, $_FILES['file']['tmp_name']);
finfo_close($finfo);
if (!in_array($type, ['image/png'])) {
  error('Uploaded file is not PNG format.');
}

// check file width/height
$size = getimagesize($_FILES['file']['tmp_name']);
if ($size[0] > 256 || $size[1] > 256) {
  error('Uploaded image is too large.');
}
if ($size[2] !== IMAGETYPE_PNG) {
  // I hope this never happens...
  error('What happened...? OK, the flag for part 1 is: <code>' . getenv('FLAG1') . '</code>');
}

// ok
$filename = bin2hex(random_bytes(4)) . '.png';
move_uploaded_file($_FILES['file']['tmp_name'], UPLOAD_DIR . '/' . $filename);

$session->set('avatar', $filename);
flash('info', 'Your avatar has been successfully updated!');
redirect('/');

看一遍代码可以发现奇怪的地方，在对上传的文件进行类型判别的时候，第一次使用的是finfo_open，第二次使用的是getimagesize。而我们要拿到flag的条件是第一次被判定为png图片，第二次被判定不是png类型。所以问题就在于这两个函数的差异判断。

可以看到，finfo_file可以识别这个图片文件，但是getimagesize不行，所以上传这个文件就可以了。

0x02 Avatar Uploader2

比较有意思的是这个第二题，首先题目自定义了个session相关的类

<?php
class SecureClientSession {
  private $cookieName;
  private $secret;
  private $data;

  public function __construct($cookieName = 'session', $secret = 'secret') {
    $this->data = [];
    $this->secret = $secret;

    if (array_key_exists($cookieName, $_COOKIE)) {
      try {
        list($data, $signature) = explode('.', $_COOKIE[$cookieName]);
        $data = urlsafe_base64_decode($data);
        $signature = urlsafe_base64_decode($signature);
    
        if ($this->verify($data, $signature)) {
          $this->data = json_decode($data, true);
        }
      } catch (Exception $e) {}
    }
  
    $this->cookieName = $cookieName;
  }

  public function isset($key) {
    return array_key_exists($key, $this->data);
  }

  public function get($key, $defaultValue = null){
    if (!$this->isset($key)) {
      return $defaultValue;
    }

    return $this->data[$key];
  }

  public function set($key, $value){
    $this->data[$key] = $value;
  }

  public function unset($key) {
    unset($this->data[$key]);
  }

  public function save() {
    $json = json_encode($this->data);
    $value = urlsafe_base64_encode($json) . '.' . urlsafe_base64_encode($this->sign($json));
    setcookie($this->cookieName, $value);
  }

  private function verify($string, $signature) {
    return password_verify($this->secret . $string, $signature);
  }

  private function sign($string) {
    return password_hash($this->secret . $string, PASSWORD_BCRYPT);
  }
}

这里的cookie值是在save函数中定义的

将传入的data转成json类型，然后value值分为两部分

第一部分是 明文的json转为base64类型

第二部分是 明文的json经过sign函数加密再转为base64类型

我们登录题目 解密一下它给的cookie

后面的是一串看不懂的字符，不过没关系，我们看看它的加密方法

  private function sign($string) {
    return password_hash($this->secret . $string, PASSWORD_BCRYPT);
  }

传入的string经过 password_hash处理，而这个password_hash有一个问题，就是只能处理72个字符，多余的会忽略掉。

也就是说如果当前session的字符超过了72个，那么它后面无论添加什么，加密出来的字符都是一样的。说这个有什么意义呢？

session中定义了一个verify函数，并且在__construct函数中用到

  private function verify($string, $signature) {
    return password_verify($this->secret . $string, $signature);
  }

public function __construct($cookieName = 'session', $secret = 'secret') {

构造函数的意思就是，将我们的cookie拿出来，通过.分割为两部分，再进行base64解码

然后再判断 明文data和经过加密的值是否匹配。

所以如果我们第一次构造出一个超过72字符的明文data，我们无论加什么字符再后面把它拼回去，都能和经过加密的值匹配上。

那这又有什么用呢？

在index.php中有一个include包含文件，既然有拼接，就会有漏洞存在，我们跟进session的get方法看看。

  public function get($key, $defaultValue = null){
    if (!$this->isset($key)) {
      return $defaultValue;
    }

    return $this->data[$key];
  }

传入的key是 theme ，那么也就是说如果我们控制了 data[‘theme’] 就可以控制任意文件包含。刚好我们可以上传文件，我们可以上传一个phar文件包含一个插有恶意代码的css文件。就可以实现rce。

data就是我们的session明文部分，只要我们先让服务器生成一个大于72个字符的session，记录下此时的加密部分，然后在session明文部分的最后加上我们的theme=>上传的文件名。就可以过掉验证，实现文件包含。
